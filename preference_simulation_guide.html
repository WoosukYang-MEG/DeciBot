Architecture & Implementation Guide

Multi-Agent Preference Simulation System

본 문서는 설계된 아키텍처 요구사항에 따라 Dynamic Interaction Network를 기반으로 구현된 Preference Simulation System의 핵심 동작 원리와 에이전트 명세(Agent Specifications)를 설명합니다. 각 에이전트는 고정된 파이프라인이 아닌 Orchestrator의 통제 하에 실시간으로 상호작용하며 최종 선호도(Preference Response)를 도출합니다.

1. System Architecture & Agent Definitions

시스템은 전체 의사결정 에피소드를 통제하는 1개의 Orchestrator Agent와, 서로 다른 인지적 역할을 수행하며 동적으로 상호작용하는 5개의 Functional Agents로 구성되어 있습니다.

2. Execution Flow: 3-Phase Architecture

본 시스템은 고정된 파이프라인(Rigid step-by-step pipeline)을 거부하고, 페르소나와 동적 맥락에 기반한 상호작용 네트워크를 구축하기 위해 세 가지 Phase로 실행됩니다.

Phase 1
Orchestrator Initialization

Orchestrator Agent가 페르소나, 동적 맥락, 태스크(Feature)를 종합적으로 분석하여 Decision Frame을 수립합니다. 상황의 특성에 따라 가장 심리학적 타당성이 높은(Psychological realism) 첫 번째 기능적 에이전트를 동적으로 결정하고 초기 가중치를 설정합니다.

Phase 2
Dynamic Interaction Network (Core Loop)

5개의 기능 에이전트 간의 재귀적 상호작용(Recursive influence)이 일어나는 핵심 구간입니다. 한 에이전트의 출력이 완료될 때마다 Orchestrator가 개입하여 다음 라우팅 경로를 통제합니다. 모든 5개 에이전트가 최소 1회 이상 고유의 출력을 기여(Distinct outputs)하도록 보장됩니다.

Phase 3
Preference Integration & Scoring

충돌 해소 및 상호작용이 수렴(Convergence)되면 Orchestrator가 최종 평가 리포트를 생성합니다. 통합된 Evaluative state를 기반으로 Task-oriented Preference Score (0~100)와 Label을 산출하며, 이 결과는 페르소나, 맥락, 감정, 태스크의 인과적 정합성(Causal coherence)을 철저히 반영합니다.

3. Implementation Details: Dynamic Routing Protocol

요청하신 Interaction Principle을 구현하기 위해, 시스템은 내부적으로 다음과 같은 라우팅 프로토콜을 수행합니다.

💡 Dynamic Interaction Network의 제어 방식

요구사항에 명시된 바와 같이, 5개의 기능 에이전트는 결코 고정된 선형 파이프라인(Fixed sequence)으로 동작하지 않습니다. Orchestrator는 매 턴마다 누적된 Debate History를 평가하여 다음과 같은 재귀적 영향(Recursive influence)을 동적으로 구현합니다.

  • Cognition이 완전히 안정화되기 전에 Perception이 Emotion을 먼저 활성화하는 경우 (예: 높은 스트레스 맥락)
  • 초기의 긍정적 Affective response를 Cognition의 트레이드오프 분석이 압도(Override)하는 경우
  • Goal의 동기 우선순위가 다시 Perception의 주의 집중(Attention priority) 방향을 재조정하는 경우
  • Cognition과 Emotion 간의 Conflict가 감지되었을 때, Intention의 확립이 지연(Hesitation)되는 경우

Routing Step Analysis (1 Cycle)

Phase 2에서 Orchestrator가 제어하는 단일 루프의 논리적 흐름입니다.

1
Agent Output Generation
이전 턴에서 Orchestrator에 의해 지목된 에이전트(Target Agent)가 누적된 컨텍스트와 자신의 System Prompt를 기반으로 고유의 출력 요건(Output requirements)을 생성합니다. 일반적인 요약(Generic summaries)을 지양하고 페르소나 특성에 기반한 구체적 판단을 출력합니다.
2
State & History Update
생성된 에이전트의 응답이 시스템의 글로벌 컨텍스트(Debate History)에 병합됩니다. 이는 후속 에이전트들이 이전 에이전트의 상태 변화(예: Emotion state shift)를 인지하고 상호작용하기 위한 기반 데이터가 됩니다.
3
Orchestrator Conflict Detection & Weighting
Orchestrator가 새롭게 업데이트된 컨텍스트를 분석하여, 에이전트 간의 Major conflicts (의견 충돌 및 모순)를 탐지하고 현시점에서 가장 강력한 영향력을 행사하는 에이전트의 가중치를 재조정합니다.
4
Next Agent Dispatch (Structured Output)
충돌 해소(Conflict resolution) 및 선호 상태 도출을 위해 가장 논리적으로 타당한 다음 에이전트를 결정합니다. API 응답의 파싱 오류를 방지하기 위해 {"next": "Target_Agent", "prompt": "Context_specific_instruction"} 형태의 JSON Schema 규격을 강제하여 시스템 실행의 안정성을 보장합니다.
